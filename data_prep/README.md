# Data Preparation

## Sources

### 1. Hand-crafted seed (`data/handcrafted_seed.jsonl`, n=10)

Designed for diversity and edge-case coverage. Created Day 1.
First-person, dense (avg ~8 fields filled).

### 2. HiNER conversion (`data/from_ner_v1.jsonl`, n=6,422)

Converted from AI4Bharat's [cfilt/HiNER-collapsed](https://huggingface.co/datasets/cfilt/HiNER-collapsed).
Real Hindi sentence distribution. Third-person, sparse (every entry has
`full_name` + ≥1 location field). Day 2 deliverable.

> HiNER's documented annotation issues — party names and common nouns
> mistagged as PER, multi-word locations truncated to a single token — make
> raw extraction noisy. The pre-training filter pass below removes the worst
> offenders so the model receives the real-Hindi distributional signal
> without learning to copy obvious mislabels into `full_name`.

**Conversion rules** (see [`ner_to_schema.py`](ner_to_schema.py)):

- Keep sentences with at least one PER entity, length 5–50 tokens
- First PER token span → `full_name`
- LOCATION entities classified via [`indian_states.py`](indian_states.py):
  - Match in `MAJOR_CITIES_HINDI` → `village_or_town`
  - Match in `INDIAN_STATES_HINDI` → `state`
  - Otherwise → `district` (conservative default)
- Only the first unfilled location field of each type is assigned
- ORG entities ignored entirely (HiNER over-tags `I-ORG` ~10:1 vs `O`)
- Trailing punctuation stripped from extracted spans
- Every entry Pydantic-validated against `CitizenProfile` before writing

**Rejection rules** (the filter pass):

1. **Length filter**: reject sentences with <5 or >50 tokens
2. **No PER**: reject sentences with no PERSON entity
3. **Short single-token name**: reject if `full_name` is a single token
   shorter than 4 Unicode codepoints (overwhelmingly common Hindi particles
   mistagged as PER)
4. **Noise stoplist match**: reject any entry where `full_name`, `state`,
   `district`, or `village_or_town` exactly matches an item in
   [`ner_stoplist.py`](ner_stoplist.py). The stoplist covers known
   problem patterns: political party names (भाजपा, कांग्रेस, etc.),
   common Hindi nouns mistagged as PER (विशेष, गंभीर, अदालत, etc.),
   and generic place nouns mistagged as LOC (देश, राज्य, जिला, etc.)
5. **Too sparse**: reject entries with `fields_filled < 2` — entries with
   only `full_name` and no location teach "extract a name" rather than
   "extract a citizen profile"

**Field coverage in `from_ner_v1.jsonl` (n=6,422):**

| Field             | Count   | %       |
|-------------------|---------|---------|
| full_name         | 6,422   | 100.0%  |
| district          | 6,396   | 99.6%   |
| state             | 34      | 0.5%    |
| village_or_town   | 16      | 0.2%    |
| (all other 17)    | 0       | 0.0%    |

The very low `state` / `village_or_town` counts reflect HiNER's habit of
truncating multi-word locations (e.g., labeling only "प्रदेश" as `B-LOC`
in "उत्तर प्रदेश"), so they don't match the canonical state list and fall
through to `district`. This is a HiNER labeling artifact, not a parser bug.

## Known limitations

These are intentionally surfaced, not silently filtered:

1. **Residual HiNER noise (post-filter).** The stoplist-based filter catches
   the highest-volume mistags, but a fraction of remaining `full_name`
   values are still city/team/institution names rather than actual
   persons (e.g., "गंगाराम" from "गंगाराम अस्पताल" = Ganga Ram Hospital;
   "राजस्थान" from "राजस्थान रॉयल्स" = Rajasthan Royals cricket team).
   Estimated residual noise rate: 10–20%. If Day 3 calibration reveals
   this hurts more than it helps, extend the stoplist with additional
   patterns or add a "PER must appear with honorific" rule.

2. **Token-level spaces around punctuation.** `input_text` is generated by
   `" ".join(tokens)`, so it includes spaces before commas, periods, danda
   (`।`), etc. The fine-tuning tokenizer handles this; flagging for
   transparency.

3. **Third-person perspective + domain mismatch.** HiNER sentences are
   primarily news-domain ("X did Y in Z"), not first-person citizen
   self-descriptions. We're using HiNER for *real Hindi distributional
   signal*, not for the task itself — synthetic generation in Days 3-7
   carries the heavier task-shaping load.

4. **Sparsity.** Every entry falls in the `sparse (≤6 fields)` bucket;
   most have exactly 2 fields filled. Dense profiles come from the
   hand-crafted seed and synthetic generation.

## Setup notes

- The `cfilt/HiNER-collapsed` dataset uses a legacy custom loader script.
  This requires `datasets>=2.18,<3` (newer versions removed script-based
  dataset support) and `trust_remote_code=True` at the `load_dataset()` call.
- Dataset is cached to `~/.cache/huggingface/hub/datasets--cfilt--HiNER-collapsed`
  (~75 MB).
- On Windows, run all scripts with `$env:PYTHONIOENCODING="utf-8"` so
  Devanagari renders correctly in stdout.

### 3. Synthetic dense (`data/synthetic_dense_v1.jsonl`, n=800)

First-person, dense citizen profiles in Devanagari Hindi. Generated by Claude
Code following the persona matrix at [`personas_full.json`](personas_full.json)
(800 slots) with reference to [`india_domain_reference.md`](india_domain_reference.md)
for surname×caste×region compatibility and welfare-wage realism.

**Composition (16 batches):**

- Batch 01 (n=50): Day 3 calibration batch — slots 1–50
- Batches 02–04 (n=150): Day 4 Session 1 — slots 51–200
- Batches 05–08 (n=200): Day 4 Session 2 — slots 201–400
- Batches 09–12 (n=200): Day 4 Session 3 — slots 401–600
- Batches 13–16 (n=200): Day 4 Session 4 — slots 601–800

**Generation methodology:**

- Persona matrix specifies demographic dimensions per slot (region, age, gender,
  occupation, income tier, caste, religion, marital status)
- Religion distribution targets ~80/12/5/3/1/1 (Hindu/Muslim/Christian/Sikh/Buddhist/Jain)
- Caste distribution targets ~30/40/15/10/5 (general/OBC/SC/ST/unspecified)
- Density target 12.0–13.0 fields per entry (refined from 11–16 in Day 3 after calibration)
- Pincode coverage 14–22% (actual coverage)
- Surname × caste × region compatibility cross-checked against domain reference
- Welfare wages (MGNREGA, ASHA, anganwadi) constrained to real government rates
- Geography-bound occupations (tea garden, salt pan, apple orchard) restricted to
  realistic regions
- First-person Devanagari, 200–400 chars per input_text
- Opener variety enforced: ≥30% non-"मेरा नाम" openers
- All entries Pydantic-validated via [`save_synthetic_batch.py`](save_synthetic_batch.py)
- Aggregated via [`aggregate_synthetic.py`](aggregate_synthetic.py)

**Calibration history:**

- Day 3 calibration (n=50) → review identified 5 refinements
- Session 1 (n=150) → review identified 3 domain errors and a need for cultural reference
- Session 2 (n=200) → introduced `india_domain_reference.md`, error rate dropped sharply
- Session 3 (n=200) → ≤1 minor flag per 12 spot-checked samples
- Session 4 (n=200) → expanded Buddhist demographics in domain reference, ran final QA

**Known limitations:**

- Tail-distribution personas (e.g., upper-caste in menial work, female bus driver
  in Buddhist context) appear at low frequency. These are not errors but
  unusual statistical outliers — present in real Indian society but rare.
- Density distribution is slightly bimodal around 12–13 rather than smoothly
  uniform 11–14, reflecting generator's natural tendency to over-include
  rather than under-include fields.
- Pincode coverage skews toward urban/elite personas; rural personas rarely
  include pincode (matches real form-filling behavior).
- bank_name fill rate is ~40% of entries with `has_bank_account: true`. This
  reflects realistic respondent behavior — many speakers say "बैंक खाता है"
  without naming the bank. The conditional extraction rule the model learns
  (extract bank_name iff mentioned in input) is correct regardless.

### 4. Hard cases (`data/hardcases_v1.jsonl`, n=40)

Hand-crafted hard cases that stress-test the model on extraction edge cases not
covered by synthetic data. Each entry targets a specific extraction difficulty,
paired with the JSON that *correct* extraction logic should produce per the
established rules.

**Categories (4 entries each):**
- Self-correction / disfluency
- Conflicting information
- Approximate values
- Negation / explicit denial
- Partial / trailing information
- Implicit information (must NOT infer)
- Multiple persons in input (extract speaker only)
- Place name variants (use speaker's wording)
- Indirect self-reference
- Mixed scripts / numerals

**Role in training:** These entries are evaluation-critical. They're small in
volume but disproportionately important — they're what reveals whether the
fine-tuned model handles real human messiness rather than only clean synthetic
inputs. Some should be held out into the validation/test split.

## 5. Train/Val/Test Split (`data/{train,val,test}_v1.jsonl`)

Week 1 final corpus assembly. Combines all four sources into stratified splits.

**Composition:**

| Source | Train | Val | Test | Total |
|---|---|---|---|---|
| Hand-crafted seed | 10 | 0 | 0 | 10 |
| HiNER (subsampled to 2000) | 1600 | 200 | 200 | 2000 |
| Synthetic dense | 640 | 80 | 80 | 800 |
| Hard cases | 20 | 10 | 10 | 40 |
| **Total** | **2270** | **290** | **290** | **2850** |

**Split design rationale:**

- **HiNER subsampled to 2000 (from 6422)**: at full ratio, sparse HiNER would
  dominate the loss signal and dilute the model's learning of dense first-person
  extraction. Subsample preserves real-Hindi distributional signal without
  overwhelming. *Note on stratification:* the brief targeted strata 600/800/600
  across `fields_filled` 2/3/4, but HiNER's actual post-filter distribution is
  `{2: 6398, 3: 24, 4: 0}` — only one meaningful stratum exists. The subsampler
  takes all 24 from stratum-3 and tops up from stratum-2 to reach the 2000 target.
- **Hard cases over-represented in val/test (25% each vs ~1.5% corpus weight)**:
  hard cases are the most informative eval signal. Balanced per category —
  every one of the 10 categories appears in val AND test (2 train / 1 val / 1 test).
- **Hand-crafted seed all to train**: too few (10) to split meaningfully; serves
  as anchor examples for the model.
- **Synthetic and HiNER**: standard 80/10/10.

**Reproducibility:** all splits use `random.seed(42)`. Re-running
[`split_corpus.py`](split_corpus.py) produces identical splits.

**Source tagging:** every entry's `metadata.source` is canonicalised to one of
`handcrafted_seed`, `hiner`, `synthetic_dense`, `hardcases`. The original
per-batch source (e.g., `synthetic-claude-day3`, `hardcases-day5`) is preserved
under `metadata.original_source` for provenance.

**See also:** [`data/split_manifest.json`](../data/split_manifest.json) (exact
entry counts per source per split),
[`split_comparison.md`](split_comparison.md) (distribution comparison across splits).
